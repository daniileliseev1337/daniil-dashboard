# История платежей по проектам + сводная панель (№2) — дизайн

Дата: 2026-06-08
Статус: согласован владельцем (brainstorming), готов к writing-plans.

## Контекст и цель

Сейчас оплата проекта хранится одним накопительным числом `projects.paid_amount`
(без даты и без истории). Доли участников (`project_shares`, кластер #1/#2)
распределяют оплату **пропорционально**: `received = paid_amount × amount / contract_sum`.

Две проблемы из эксплуатации:
- **№3 (доработка):** вкладка «Финансы» (помесячный кошелёк по дате транзакции) не
  может показать проектные деньги — у `paid_amount` нет даты платежа. Владелец хочет,
  чтобы «Доходы» совмещали ручные транзакции и полученное по проектам.
- **№2:** нет сводной «сколько и за что получено»; нужна динамическая панель на вкладке
  «Проекты» — выбираешь проекты, она суммирует.

Решение: ввести **историю платежей** (`project_payments`, дата + сумма). `paid_amount`
становится производным (`SUM(платежей)`), пропорциональная модель долей **не меняется**,
появляется дата для помесячного разреза Финансов и детализация для сводной панели.

## Согласованные решения (brainstorming 2026-06-08)

1. **Платёж — в счёт договора, пропорционально** (не адресный участнику). Модель долей
   остаётся прежней.
2. **`paid_amount = SUM(платежей)`**, поддерживается **БД-триггером** (источник правды —
   платежи; все текущие чтения `paid_amount` работают без изменений).
3. **Backfill:** существующие `paid_amount > 0` → один стартовый платёж, `paid_on = сегодня`
   (дата переноса). Владелец принял, что текущий месяц Финансов раздуется.
4. **Scope:** история платежей + панель №2 в одном дизайне (план разбивает на юниты).
5. **Панель №2** показывает: Получено (моя доля), Остаток/к получению, Сумма договоров,
   разбивку по каждому выбранному проекту.

## Архитектура по секциям

### 1. Модель данных

Таблица `public.project_payments`:
- `id uuid primary key default gen_random_uuid()`
- `project_id uuid not null references public.projects(id) on delete cascade`
- `amount numeric not null check (amount > 0)`
- `paid_on date not null`
- `note text`
- `created_at timestamptz not null default now()`
- `created_by uuid references auth.users(id) default auth.uid()`
- индекс по `project_id`.

**RLS:** читать/писать платежи может **только владелец проекта**
(`exists (select 1 from projects p where p.id = project_id and p.owner_id = auth.uid())`).
Участник детально платежи и сумму договора не видит — его проекция остаётся через
`get_my_shares` (агрегат по `paid_amount`). Это сохраняет приватность.

**Триггер `recalc_project_paid_amount`** (AFTER INSERT OR UPDATE OR DELETE FOR EACH ROW,
SECURITY DEFINER, `set search_path = public, pg_temp`):
пересчитывает `projects.paid_amount = COALESCE(SUM(amount), 0)` для затронутого проекта.
Обрабатывает и `NEW.project_id`, и `OLD.project_id` (на случай переноса платежа между
проектами и DELETE). FOR EACH ROW — надёжно для любых правок, включая прямые одиночные;
при bulk через RPC сработает несколько раз (платежей у проекта немного — приемлемо).

### 2. Backfill (миграция данных)

```
insert into project_payments (project_id, amount, paid_on, note, created_by)
select p.id, p.paid_amount, current_date, 'Перенос из paid_amount', p.owner_id
from projects p
where p.paid_amount > 0
  and not exists (select 1 from project_payments pp where pp.project_id = p.id);
```
Идемпотентно (`not exists`). Триггер пересчитает `paid_amount` в то же значение — потерь нет.

### 3. RPC `set_project_payments`

`set_project_payments(p_project_id uuid, p_rows jsonb)` — SECURITY DEFINER, `search_path`
фикс, гейт «владелец проекта». Replace-all в одной транзакции (delete по `project_id` +
insert новых строк из jsonb), как `set_project_shares`. Триггер обновит `paid_amount`.
GRANT execute to authenticated.

### 4. UI ввода платежей (ProjectForm, `src/App.jsx`)

Поле «Оплачено факт» (`paidAmount`, ручной ввод) заменяется секцией **«Платежи»**:
- список строк: дата (`paid_on`) + сумма (`amount`) + заметка (`note`), кнопки +добавить/удалить;
- итог «Оплачено всего: SUM» — read-only (отражает будущий `paid_amount`);
- сохранение проекта вызывает `set_project_payments` (атомарно), затем рефетч.
- При установке стадии «Оплачен» — предложить добавить платёж на остаток
  (`contract_sum − текущая сумма платежей`) датой сегодня (сохраняем удобство прежнего
  автозаполнения paid=contract из кластера #1/#2, но уже как явный платёж).

`paidAmount` в JS-модели проекта остаётся (производное от БД) — карточки/отчёты/аналитика,
читающие `p.paidAmount`, не трогаем.

**Важно (риск рассинхрона):** `projectJsToDb`/`updateProject` сейчас пишет `paid_amount`
из формы. Так как формой он больше не управляется, из `projectJsToDb` поле `paid_amount`
нужно **убрать** (не отправлять в `update projects`) — иначе сохранение проекта затрёт
значение, выставленное триггером. Источник правды по `paid_amount` — только триггер.

### 5. Finance — помесячное вливание (`src/App.jsx`, `dashboardMetrics.js`)

App загружает платежи моих проектов в state `paymentsByProject` (как `sharesByProject`),
передаёт в `Finance`. Карточка **«Доходы»** за выбранный месяц (`monthF`):
```
Доходы = (ручные income за месяц, как сейчас)
       + Σ по платежам МОИХ проектов с paid_on в этом месяце:
            payment.amount × ownerShareAmount(project, shares) / contract_sum
```
(пропорция = моя доля владельца; согласовано с `ownerReceived`). Считается **на лету**,
строки в `transactions` НЕ создаются (нет дублей). «Баланс» = Доходы − Расходы.
Проектные доходы — отдельной категорией «Проектные доходы» в пироге «Источники доходов».

**Ограничение приватности:** помесячно вливаются только **мои проекты** (где `ownerId = я` —
есть доступ к датам платежей). Мои доли в **чужих** проектах остаются агрегатом в блоке
«По проектам» (нет доступа к датам чужих платежей по RLS). Это явное и приемлемое ограничение.

Блок «По проектам · моя доля» (сделан в №3) сохраняется; «Получено» там = из `paid_amount`
(= сумма платежей), значение не меняется.

### 6. №2 — сводная панель (вкладка «Проекты», `src/App.jsx`)

- Режим выбора: чекбоксы на карточках проектов (тумблер «Выбрать»/сброс).
- Сворачиваемая sticky-панель по выбранным проектам:
  - **Получено** (моя доля) = Σ `proportionReceived(paid_amount, ownerShareAmount, contract)`;
  - **Остаток** = Σ (`ownerShareAmount − received`);
  - **Сумма договоров** = Σ `contract_sum`;
  - **Разбивка по каждому**: список «проект → моя доля получено».
- Считается из `projects` + `sharesByProject` (как дашборд) — детальные платежи не нужны.

### 7. Расчёты + тесты (`dashboardMetrics.js`, `dashboardMetrics.test.js`)

Новые чистые функции (без React/Supabase):
- `myProjectIncomeForMonth(paymentsByProject, projects, sharesByProject, ownerId, monthStr)`
  → моя доля проектных платежей за месяц `YYYY-MM`.
- `selectionTotals(selectedProjects, sharesByProject, ownerId)`
  → `{ received, remaining, contract, breakdown: [{name, received}] }`.
Покрыть vitest (граничные: нет платежей, доля 0, contract 0, месяц без платежей).

## Декомпозиция на юниты (для writing-plans)

1. **БД:** таблица + RLS + триггер + `set_project_payments` + backfill-миграция; verify-rls скрипт.
2. **Расчёты+тесты:** `myProjectIncomeForMonth`, `selectionTotals` + vitest.
3. **UI платежей:** секция «Платежи» в ProjectForm + сохранение через RPC + загрузка `paymentsByProject`.
4. **Finance-вливание:** проектные доходы моей доли по месяцу в «Доходы» + категория.
5. **Панель №2:** режим выбора на карточках + сводная панель.

Юниты 3–5 зависят от 1–2. Юниты 1 и 2 независимы между собой.

## Риски и ограничения

- Backfill `paid_on = сегодня` раздувает текущий месяц Финансов (принято владельцем).
- Помесячные проектные доходы — только для моих проектов (приватность чужих платежей).
- Триггер FOR EACH ROW при bulk-replace срабатывает несколько раз (платежей мало — ок).
- `paid_amount` остаётся колонкой (денормализация) — единственный писатель кроме триггера
  быть не должен; ручной ввод `paidAmount` из ProjectForm убирается (теперь только платежи).

## Критерии успеха / верификация

- Миграции применяются к живой БД; `verify` под двумя юзерами → RLS_OK (участник не видит
  чужие платежи; владелец видит/пишет свои).
- После добавления/удаления платежа `paid_amount` = сумме платежей (триггер).
- Backfill: `paid_amount` проектов не изменился; у каждого с оплатой появился 1 платёж.
- Доли/дашборд/`get_my_shares` дают прежние значения (пропорция не сломана).
- Finance: «Доходы» за месяц = ручные income + моя доля платежей моих проектов того месяца.
- Панель №2: суммы по выбранным проектам совпадают с дашбордом/блоком «По проектам».
- `npm run build` зелёный; vitest (старые 55 + новые) зелёный.
