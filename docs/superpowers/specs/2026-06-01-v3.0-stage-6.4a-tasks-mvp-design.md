# Этап 6.4a — Слой задач (MVP). Дизайн

> Дата: 2026-06-01. Статус: согласован с владельцем, готов к writing-plans.
> ТЗ v3.0 раздел 4.2. Этап 6.4 разбит на **6.4a (MVP, этот документ)** и
> **6.4b (версионирование «ТЗ задачи» + diff-просмотр + апрув-механика)**.

## 1. Цель и охват

Задача — **самостоятельная сущность** (не подзадача проекта): может существовать
без привязки к проекту. Слой задач — глобальная надстройка, видящая задачи по
всем доступным проектам сразу, с фильтрами. Плюс «проваливание» из карточки
проекта в его задачи.

**В охвате 6.4a:** сущность задач, RLS, RPC списка, глобальная вкладка
(список+фильтры, доска-переключатель), секция задач в карточке проекта,
быстрое создание, принятие в работу/смена статуса/редактирование/удаление,
Telegram-уведомления.

**НЕ в охвате 6.4a (уходит в 6.4b):** версионирование поля «описание/ТЗ»,
diff-просмотр изменений, апрув-механика изменений и статусов.

**Подход:** прагматичный MVP на refetch (как сейчас работают проекты/комментарии).
**Realtime — обязательный пункт roadmap**, не выбрасывается: БД-сторона готовится
сразу (см. §6), живая подписка включается в 6.4b без новой миграции и без
переделки MVP.

## 2. Данные

### Таблица `project_tasks`

| Колонка | Тип | Назначение |
|---|---|---|
| `id` | uuid PK default gen_random_uuid() | |
| `project_id` | uuid NULL → projects(id) ON DELETE SET NULL | опц. привязка; удалили проект → задача становится личной |
| `author_id` | uuid NOT NULL → profiles(id) | создатель |
| `assigned_to` | uuid NULL → profiles(id) | исполнитель |
| `title` | text NOT NULL | заголовок |
| `description` | text | «ТЗ задачи» (в 6.4b версионируется) |
| `status` | text NOT NULL default 'Новая' | CHECK in (Новая, В работе, На проверке, Готово, Отменена) |
| `priority` | text NOT NULL default 'Обычный' | CHECK in (Низкий, Обычный, Высокий) |
| `due_date` | date NULL | дедлайн |
| `sort_order` | int NOT NULL default 0 | порядок внутри колонки доски (drag-drop) |
| `created_at` | timestamptz NOT NULL default now() | |
| `updated_at` | timestamptz NOT NULL default now() | через существующий триггер `touch_updated_at` |

Значения `status`/`priority` — русские строки, консистентно с `projects.stage`.

Индексы: `project_id`, `assigned_to`, `status`, `due_date`.

## 3. Права (RLS)

Предикат доступа: «от проекта, иначе личная». Переиспуёт существующие хелперы
(`can_access_project_comments`, `is_project_editor`, `is_project_owner`,
`is_admin`, `is_approved`).

- **SELECT (visibility):**
  ```
  (project_id IS NOT NULL AND can_access_project_comments(project_id))
  OR (project_id IS NULL AND (author_id = auth.uid() OR assigned_to = auth.uid() OR is_admin()))
  ```
- **INSERT (with_check):**
  ```
  is_approved() AND author_id = auth.uid()
  AND (project_id IS NULL OR can_access_project_comments(project_id))
  ```
- **UPDATE (using И with_check одинаковы):**
  ```
  author_id = auth.uid() OR assigned_to = auth.uid() OR is_admin()
  OR (project_id IS NOT NULL AND is_project_editor(project_id))
  ```
  with_check дополнительно гарантирует доступность нового `project_id`
  (нельзя увести задачу в чужой проект): `(project_id IS NULL OR can_access_project_comments(project_id))`.
- **DELETE (using):**
  ```
  author_id = auth.uid() OR is_admin()
  OR (project_id IS NOT NULL AND is_project_owner(project_id))
  ```
  Исполнитель удалять не может.

## 4. Доступ к данным (RPC + прямые мутации)

**Нюанс:** RLS на `profiles` отдаёт только свой профиль, поэтому embed имён
автора/исполнителя через FK не работает для чужих — ровно поэтому
`get_project_comments`/`get_project_files` сделаны `SECURITY DEFINER`.

- **RPC `get_tasks(p_project_id uuid default null, p_status text default null, p_assigned_to uuid default null)`**
  — `SECURITY DEFINER`, STABLE. **Внутри сам дублирует предикат доступа** (обязательно, раз
  обходит RLS) и возвращает задачи + `project_name`, `author_name`, `assignee_name`.
  Покрывает и глобальную вкладку, и секцию в проекте (через `p_project_id`).
  Фильтры опциональны (NULL = не фильтровать). Сортировка по `status`, затем
  `sort_order`, затем `created_at`.
- **Мутации (create/update/delete)** — прямой `client.from('project_tasks')` под
  RLS (политики §3 всё проверяют). `insert ... .select('id').single()`, после —
  refetch через `get_tasks`.

## 5. UI

Адаптеры `taskDbToJs` / `taskJsToDb` (snake↔camel, по образцу `projectDbToJs`).

### Вкладка «Задачи» (новый таб в `TABS`, иконка `ListTodo`)
- Шапка: переключатель **[Список | Доска]**, кнопка **«+ Новая задача»**.
- Фильтры: проект ▾, статус ▾, исполнитель ▾, срок ▾, тумблер «только мои».
- **Список (основной режим):** таблица — бейдж статуса, заголовок, проект,
  исполнитель, приоритет, срок. Клик по строке → модалка задачи. Сортировка по
  сроку/приоритету/дате.
- **Доска (переключатель):** колонки по статусам, карточки; перетаскивание между
  колонками = смена статуса (порядок — `sort_order`). Drag-drop на **framer-motion**
  (уже в зависимостях, новых библиотек не добавляем). Fallback при сложности —
  смена статуса из карточки. «Отменена» — свёрнутая/скрытая колонка.

### Модалка задачи (по образцу `ProjectForm`)
Поля: заголовок, описание («ТЗ»), проект (селект, пусто = личная), исполнитель
(селект), статус, приоритет, срок; для существующей — автор + даты. Кнопки
Сохранить/Удалить (видимость по правам).

**Источник списка исполнителей:** для проектной задачи — участники проекта
(`get_project_members`) + владелец; для личной (без проекта) — поле «исполнитель»
по умолчанию пустое/сам автор (полный справочник пользователей не показываем, это
не нужно в MVP). При смене проекта список исполнителей пересобирается.

### Секция «Задачи проекта» в карточке проекта (`ProjectForm`)
Рядом с файлами/комментариями: компактный список задач проекта (бейдж статуса +
заголовок + исполнитель) + инлайн быстрое добавление (заголовок + Enter, привязка
к проекту автоматом). Клик → та же модалка задачи.

### Точка расширения под Realtime (6.4b)
Список задач держится в одном состоянии, наполняемом из `get_tasks`. В 6.4b
добавляется `supabase.channel('project_tasks')`-подписка, которая на событие
делает мёрж/refetch. MVP-архитектура это уже допускает — переделок не требует.

## 6. Telegram-уведомления

Переиспуём бот и обёртку `sendTelegramNotify`. Расширяем Edge Function
`telegram-notify`:
- **Новые типы:** `task_assigned`, `task_status`, `task_created`.
- **Новый флаг `profiles.notif_task`** (default true) — добавить в
  `update_notification_settings` и в UI настроек уведомлений (один флаг на все
  события задач; можно разнести позже).
- **Резолв адресатов — внутри Edge Function** (фронт шлёт `{ type, taskId }`,
  функция под `service_role` читает задачу и определяет получателей):
  - `task_assigned` → исполнитель (`assigned_to`);
  - `task_status` → автор + исполнитель, кроме инициатора изменения;
  - `task_created` → участники проекта, кроме автора (через `project_members`/owner).
  - Перед отправкой — проверка `notif_task = true` и наличия `telegram_chat_id`.
- **Вызовы с фронта** (best-effort): после insert с проектом → `task_created`;
  после смены `assigned_to` → `task_assigned`; после смены `status` → `task_status`.

**Проверить на этапе плана:** задеплоена ли `telegram-notify` в локальном
Supabase (её код мог жить только в облаке, как было с `yandex-disk`). Если нет —
создать/задеплоить локально (как сделали с `nextcloud`).

## 7. Realtime-готовность БД (важно)

В миграцию 6.4a включить добавление таблицы в публикацию:
`ALTER PUBLICATION supabase_realtime ADD TABLE public.project_tasks;`
Это не влияет на MVP (едет на refetch), но делает Realtime включаемым на фронте в
6.4b **без новой миграции**. Так пункт «Realtime» зафиксирован в коде, а не на словах.

## 8. Критерии приёмки 6.4a

1. Можно создать задачу: личную (без проекта) и в проекте.
2. Глобальная вкладка показывает все доступные задачи; фильтры (проект, статус,
   исполнитель, срок, «только мои») работают.
3. Видимость соответствует правам §3 (личные не видны посторонним; проектные —
   по доступу к проекту); проверить под двумя разными пользователями.
4. Принятие в работу и смена статуса работают (список и доска); доска
   переключается со списком.
5. Назначение исполнителя, редактирование полей, удаление — по правам.
6. Секция задач в карточке проекта: список + быстрое создание с авто-привязкой.
7. Telegram: приходят `task_assigned`, `task_status`, `task_created` адресатам с
   `notif_task=true` и привязанным чатом; инициатор смены статуса себе не дублирует.
8. `project_tasks` в публикации `supabase_realtime` (проверка `pg_publication_tables`).
9. Сборка фронта проходит; деплой через `deploy/nextcloud/deploy-web.sh`.
