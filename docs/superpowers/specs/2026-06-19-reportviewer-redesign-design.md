# Дизайн: редизайн ReportViewer + «отчёт по выбранным» + устранение XSS

**Дата:** 2026-06-19
**Замечание:** #4 из очереди (подтверждено владельцем).
**Файл:** `src/App.jsx`, компонент `ReportViewer` (текущий ~7355–7602).

## Проблема

1. **Вёрстка.** Таблица в 8 колонок (`minWidth:760`) — на телефоне горизонтальный
   скролл; на ПК «как набросок».
2. **Дубль разметки + XSS.** Отчёт рендерится дважды одинаковым содержимым:
   (а) строкой через `innerHTML` в `#report-print-root` (для `@media print` → Ctrl+P);
   (б) безопасным JSX (экранное превью). В (а) поля `${p.name}`, `${p.client}`,
   `${p.notes}`, `${p.executor}`, `${p.type}` вставляются БЕЗ экранирования —
   **stored XSS** (M3 из аудита). Две копии надо держать синхронно.
3. **Нет «отчёта по выбранным».** Сейчас фильтр только по стадии.

## Цель

Переверстать отчёт (ПК + телефон), добавить отчёт по выбранным проектам и режим
«для заказчика», и закрыть XSS — единым presentational-компонентом без `innerHTML`.

## Решение

### 1. Архитектура: один источник разметки, XSS закрывается by design

Вынести тело отчёта в чистый presentational-компонент:

```
<ReportDocument projects={[...]} mode={"full" | "client"} meta={{title, dateStr, subtitle}} />
```

- Только JSX — React экранирует весь текст, **XSS невозможен** (узкое место `innerHTML`
  устраняется полностью, не «экранированием вручную»).
- Печать — через `@media print`: показываем этот же React-DOM (контейнер отчёта),
  прячем остальное приложение. `#report-print-root` + строковый `innerHTML` удаляются.
- Один источник правды — нет рассинхрона двух копий.

`ReportDocument` — презентационный (без supabase/состояния): на вход проекты + режим,
на выход разметка. Тестируется и читается изолированно.

### 2. Раскладка (вариант A: таблица на ПК → карточки на телефоне)

Внутри `ReportDocument`, по `useIsMobile()` (≤640px):

- **ПК (`!isMobile`):** причёсанная таблица — те же колонки, но аккуратные отступы,
  типографика, выравнивание; тёмная бренд-шапка «КЛИМАТ-ПРО».
- **Телефон (`isMobile`):** каждый проект — карточка: шапка (название + клиент),
  ниже строки «поле: значение» (Стадия, Тип, Исполнитель, По договору,
  [Оплачено, Остаток — только в полном режиме], Дедлайн). Без горизонтального скролла.
- **KPI-блок** сверху — адаптивный (на телефоне карточки в колонку/перенос).

### 3. Режимы: тумблер «Полный / Для заказчика»

Переключатель в модалке настройки отчёта (`mode`).

- **Полный (`full`):** все колонки + KPI = Сумма договоров / Получено / К получению / % оплаты.
- **Для заказчика (`client`):** скрыты колонки **Оплачено** и **Остаток**; KPI = только
  «Сумма договоров». Остаются: Проект/Клиент, Тип, Исполнитель, Стадия, По договору,
  Дедлайн, Примечания.

Режим прокидывается в `ReportDocument` (`mode`) и определяет набор колонок/KPI в обеих
раскладках (таблица и карточки).

### 4. Два входа

- **(а) Существующий:** кнопка «Экспорт отчёта» (App `setReportModal(true)`) → отчёт по
  всем проектам, фильтр по стадии в модалке (как сейчас).
- **(б) Новый — по выбранным:** на вкладке Проекты в режиме «Выбрать», в нижней
  sticky-сводке — кнопка «Сформировать отчёт» → открывает отчёт по выбранным проектам.
  Фильтр по стадии при этом скрыт (источник = ручной выбор).

  Передача выбора: `Projects` получает колбэк-проп `onMakeReport(selectedProjects)`;
  App хранит `reportProjects` (null = все; массив = выбранные) и открывает `ReportViewer`
  с этим набором. `ReportViewer` принимает опц. проп `presetProjects` (если задан — не
  показывает фильтр по стадии, использует переданный набор).

### 5. Сохранение PDF

Кнопка «🖨 Печать / Сохранить PDF» вызывает `window.print()` — работает и на ПК, и на
телефоне (на телефоне системный диалог → «Сохранить PDF»). Текущую подсказку Ctrl+P
оставляем как hint для десктопа.

### 6. Цветовая гамма и контраст (явное требование владельца)

Текст НЕ должен сливаться с фоном, **особенно в выбранных/активных состояниях**.

- Выбранный чип/тумблер: **тёмный текст** (`#0a0a0a`/`#1c1c1a`) на золотом акценте
  (`#d4af37`) — НЕ белый текст на золотом (текущая болячка: белый на светлом золоте
  читается плохо).
- Неактивные элементы: достаточный контраст (не светло-серый на тёмно-сером впритык).
- Печатная (светлая) тема отчёта: тёмный текст на белом, акценты-бейджи стадий — с
  читаемым контрастом (фон стадии полупрозрачный, текст насыщенный того же тона).
- Ориентир — WCAG AA (контраст текста ≥ 4.5:1) на глаз, без формальной проверки.

## Вне scope (YAGNI)

- Серверный экспорт PDF / шаблонизатор — печать браузером достаточна.
- Доли участников / платежи по датам в отчёте — отдельная тема.
- Сохранение пресетов отчёта, расписания, e-mail-рассылка.
- Графики/диаграммы в отчёте — это табличный отчёт.

## Верификация

- `npm run build` зелёный.
- XSS: проект с именем/примечанием вида `<img src=x onerror=alert(1)>` отображается
  как текст (не исполняется) — проверка, что `innerHTML`-пути не осталось.
- Раскладка: ПК — таблица; ширина ≤640 — карточки (через preview на узком окне).
- Режим «для заказчика»: нет колонок Оплачено/Остаток и KPI Получено/К получению.
- «По выбранным»: из режима выбора Проекты отчёт содержит ровно выбранные проекты.
- Печать: `window.print()` открывает диалог; в печати — только отчёт (приложение скрыто).
- Контраст: выбранный чип/тумблер читаем (тёмный текст на золоте).

## Затрагиваемые места (ориентир для плана)

- `src/App.jsx`: `ReportViewer` (рефактор — выделить `ReportDocument`, убрать `innerHTML`/
  `#report-print-root`, добавить `mode`, `presetProjects`, кнопку печати); `Projects`
  (кнопка «Сформировать отчёт» в sticky-сводке режима выбора + проп `onMakeReport`);
  App (state `reportProjects`, проброс в `ReportViewer`, проп в `Projects`).
- Печатный `@media print` CSS — переключить с `#report-print-root` на контейнер
  React-отчёта.
