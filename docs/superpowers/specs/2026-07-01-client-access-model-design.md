# Модель доступа заказчика «3.0» — полные вкладки со своей видимостью

> Стадия: дизайн (brainstorm 2026-07-01). Заменяет портальную модель «одна вкладка Мои заказы»
> из `2026-06-28-customer-2.0-design.md` на набор вкладок **Дашборд / Проекты / Задачи / Финансы**
> с данными заказчика, при сохранении инварианта приватности §1.

## 1. Цель

Сейчас чистый заказчик (`role client` без `employee`) видит **одну** вкладку «Мои заказы»
(`App.jsx:9144-9145`, `clientView ? [{myorders}]`). Владелец хочет, чтобы у заказчика был
**полноценный интерфейс со своими данными** — Дашборд, Проекты, Задачи, Финансы — как у сотрудника
по форме, но строго со своей видимостью (без себестоимости, долей, чужих проектов, цифр фирмы).

Портальная модель была не багом: она следствие принципа безопасности §1 (`customer-2.0-design.md`
§7) — заказчик получает данные ТОЛЬКО через узкие SECURITY DEFINER-проекции, прямой доступ к
таблицам закрыт. Эту модель НЕ ломаем — расширяем её на 4 вкладки поверх тех же безопасных проекций.

## 2. Решения (зафиксированы с владельцем, 2026-07-01)

| # | Ось | Решение |
|---|-----|---------|
| 1 | Набор вкладок заказчика | **Дашборд, Проекты, Задачи, Финансы**. НЕ даём: Заказчики (база клиентов фирмы), Заявки (входящие сотруднику), Аналитика, Admin |
| 2 | Название | Вкладка проектов заказчика называется **«Проекты»** (единообразие с сотрудником), не «Мои заказы» |
| 3 | Финансы заказчика | **Договор + оплачено + остаток + история платежей** по его заказам. СТРОГО без себестоимости, долей, маржи, чужого |
| 4 | Дашборд заказчика | Сводка по его заказам: активные заказы, ближайший дедлайн, всего оплачено/остаток, открытые задачи, блок «Требует внимания» (задачи «На проверке» → приёмка) |
| 5 | Приватность | Инвариант §1 сохраняется: данные заказчика только через SECURITY DEFINER-проекции, без прямого `from('projects').select('*')` |
| 6 | Подход к фронту | **C (гибрид):** тонкие заказчичьи контейнеры-вкладки на проекциях + реюз чистых презентационных кусков без чувствительных пропсов |

**Вне scope (YAGNI, будущие фичи — записаны в `docs/IDEAS.md`):**
- Оплата проектов исполнителям заказчиком; стоимость задач + оплата задач.
- Аналитика заказчику.
- Приватная переписка/файлы заказчику (фазы 2-3, отдельная задача).

## 3. Факты БД / кода (проверено на живом коде, 2026-07-01)

- **Роли** — тонкий слой `user_roles(user_id, role)` CHECK `employee|client|visitor`; фронт оперирует
  `myRoles` (`App.jsx`). `isClientOnly = client && !employee && role!=='admin'` (`App.jsx:9131`);
  `clientView = isClientOnly || (canSwitchView && viewMode==='client')` (`:9132`);
  `canSwitchView = client && (employee || admin)` (`:9129`) — гибрид переключает вид.
- **`get_my_client_projects()`** (миграция `20260629_0009`) — SECURITY DEFINER `stable`,
  `where clients.user_id = auth.uid()`. Возвращает: `id, name, stage, start_date, deadline,
  contract_sum, paid_amount, executor, visibility, open_task_count`. **НЕ** содержит `owner_id`,
  `notes`, доли, себестоимость. → покрывает Проекты, Финансы (договор/оплачено/остаток), Дашборд.
- **Задачи заказчика** — доступ дан: `client_task_tz_access` (`20260629_0002`), приёмка
  `client_set_task_status` (`0003`, На проверке→Готово / вернуть→В работе). Список — `get_tasks`.
- **`project_payments`** (дата+сумма, триггер `recalc_project_paid_amount`, RPC `set_project_payments`)
  — RLS сейчас **только владелец проекта** → заказчику закрыто. Для истории платежей нужна проекция.
- **Guard вкладок** — `effectiveTab` (`App.jsx:9158`) рендерит только вкладку из текущего `TABS`;
  `CommandPalette` фильтрует по `allowedTabs`/`restricted` (`:9169-9177`).
- **Чистые расчёты** — `src/lib/dashboardMetrics.js` (без React/Supabase, vitest) — образец для
  агрегатов заказчика.

## 4. Изменения БД

### 4.1. Новая проекция `get_my_project_payments()`

Единственное недостающее звено (история платежей заказчику). SECURITY DEFINER `stable`,
`set search_path = public, pg_temp`:

```sql
create or replace function public.get_my_project_payments()
returns table(project_id uuid, project_name text, paid_on date, amount numeric)
language sql stable security definer set search_path to 'public','pg_temp' as $$
  select pp.project_id, p.name, pp.paid_on, pp.amount
  from public.project_payments pp
  join public.projects p on p.id = pp.project_id
  join public.clients   c on c.id = p.client_id
  where c.user_id = auth.uid()
  order by pp.paid_on desc;
$$;
grant execute on function public.get_my_project_payments() to authenticated;
```

Поля узкие — только `project_id/name/paid_on/amount`. Категорий, себестоимости, чужих платежей нет.
`get_my_client_projects` полей для §2/§3 хватает — доп. изменений в неё не требуется.

### 4.2. Проверка RLS

`deploy/verify-client-payments-rls.sh` (по образцу `verify-shares-rls.sh`; точная папка — в плане):
под тестовым заказчиком `get_my_project_payments()` возвращает только его платежи; чужой платёж → 0 строк.

## 5. Изменения фронта (подход C)

### 5.1. Вкладки clientView (`App.jsx:9144`)

```
clientView ? [
  { id:'dashboard', label:'Дашборд',  Icon: LayoutDashboard },
  { id:'projects',  label:'Проекты',  Icon: FolderKanban },
  { id:'tasks',     label:'Задачи',   Icon: ListTodo },
  { id:'finance',   label:'Финансы',  Icon: Receipt },
] : ...
```
`effectiveTab`/`CommandPalette.allowedTabs` работают из `TABS.map(id)` — менять не нужно.

### 5.2. Bootstrap заказчика

При `clientView` грузить из проекций (НЕ из `fetchProjects`):
`get_my_client_projects()`, задачи заказчика через `get_tasks(...)` (сигнатуру заказчичьей выборки
сверить — см. §8), `get_my_project_payments()`.

### 5.3. Контейнеры-вкладки (новые, тонкие)

- **`ClientDashboard`** — KPI из проекций (активные заказы, ближайший дедлайн, всего оплачено/остаток,
  открытые задачи) + «Требует внимания» (задачи со статусом «На проверке» → кнопка приёмки).
- **`ClientProjects`** — карточки проектов из `get_my_client_projects` (стадия, дедлайн,
  счётчик задач, договор/оплачено/остаток) + «Создать проект» (существующая заявка `project_requests`);
  открытие проекта → его задачи.
- **`ClientTasks`** — задачи по заказам (доска/список, ТЗ, комментарии, приёмка Принять/Вернуть) —
  перенос функциональности из нынешнего портала во вкладку.
- **`ClientFinance`** — по каждому заказу договор/оплачено/остаток + история платежей
  (`get_my_project_payments`); суммарные оплачено/остаток. Без себестоимости/долей.

Реюз презентационных кусков (карточка проекта, доска/список задач, строка платежа) — им НЕ передаём
чувствительные пропсы. Ни один контейнер заказчика не вызывает прямой `from()`.

### 5.4. Чистые расчёты

`src/lib/clientMetrics.js` (без React/Supabase, аналог `dashboardMetrics.js`):
`clientTotals(projects)` → {активные, оплачено, остаток, открытые}; `attentionTasks(tasks)` →
задачи «На проверке»; `paymentsByProject(payments)`. Покрыто vitest.

## 6. Обработка краёв

- Нет заказов → пустое состояние «У вас пока нет заказов» + «Создать проект».
- Гибрид (`client+employee`) — `viewMode==='client'` показывает 4 вкладки заказчика, `employee` —
  полный набор; механизм переключения существующий (`canSwitchView`).
- Потеря роли `client` в рантайме → существующая логика ролей (редирект/логин).

## 7. Тестирование / критерии верификации

1. vitest на `clientMetrics.js` (агрегаты дашборда/финансов из проекций).
2. `verify-client-payments-rls.sh` → заказчик видит только свои платежи.
3. E2E под тестовым **чистым** заказчиком (`client` без `employee`, «Задача 0» — создать такой
   аккаунт): открываются 4 вкладки; данные только свои; **нигде** нет себестоимости/долей/чужих
   проектов; приёмка задачи работает; «Создать проект» создаёт заявку.
4. Регресс: сотрудник и гибрид (employee-режим) видят прежний полный набор без изменений.
