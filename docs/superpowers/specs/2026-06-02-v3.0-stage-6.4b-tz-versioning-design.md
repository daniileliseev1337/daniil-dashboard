## 1. Цель и охват

6.4a дал MVP слоя задач (сущность, RLS, RPC списка, вкладка, доска, Telegram-каркас).
6.4b закрывает оставшиеся требования ТЗ 4.2:

- **Версионирование «ТЗ задачи»** (поле `description`) с историей.
- **Diff-просмотр** изменений (что добавилось/удалилось).
- **Апрув-механика** изменений ТЗ и сдачи задачи.
- **Сценарий «есть вопросы»** — переписка/комментарии по задаче (отдельная сущность).
- **Живая Realtime-подписка** на фронте (БД-сторона готова с 6.4a).
- **Telegram-relay через VPS** — довести доставку уведомлений до рабочего состояния.

**Подход:** прагматично, минимум новых сущностей, переиспользуем существующие
паттерны (RPC `SECURITY DEFINER` как `get_tasks`/`get_project_comments`, мутации
под RLS, refetch + точечный Realtime). Без новых фронт-зависимостей.

### Согласованные развилки (brainstorming 2026-06-02)

| Вопрос | Решение |
|---|---|
| Апрув: кто/что | Двусторонний апрув правок ТЗ (автор ↔ исполнитель) + апрув сдачи (переход в «Готово» — только автор) |
| Поведение pending-версии | **Блокирующее**: новая версия `pending`, действующим остаётся последнее `approved`, пока другая сторона не апрувнет |
| Формат обсуждения | **Гибрид**: лента-чат, любое сообщение можно пометить флагом «вопрос» → попадает в счётчик открытых |
| Diff | **Построчный** (git-стиль, целые строки ±), без внешних зависимостей |
| Realtime охват | Список + доска (`project_tasks`); внутри карточки версии/комментарии — refetch |
| Telegram-relay | Полный: код через env `TELEGRAM_API_BASE` + reverse-proxy в Caddy на VPS (SSH разрешён владельцем) |

## 2. Данные

### Таблица `task_tz_versions` (версии ТЗ — snapshot полного текста)

| Колонка | Тип | Назначение |
|---|---|---|
| `id` | uuid PK default gen_random_uuid() | |
| `task_id` | uuid NOT NULL → project_tasks(id) ON DELETE CASCADE | |
| `version_no` | int NOT NULL | порядковый номер версии в рамках задачи |
| `content` | text NOT NULL | полный снимок текста ТЗ на момент версии |
| `status` | text NOT NULL default 'pending' | CHECK in (`pending`, `approved`, `rejected`) |
| `proposed_by` | uuid NOT NULL → profiles(id) | кто предложил версию |
| `resolved_by` | uuid NULL → profiles(id) | кто апрувнул/отклонил |
| `created_at` | timestamptz NOT NULL default now() | |
| `resolved_at` | timestamptz NULL | момент апрува/отклонения |

- **Действующее ТЗ** = последняя `approved`-версия. `project_tasks.description`
  хранит её текст (синхронизируется при апруве). Существующий фронт/код читает
  `description` без изменений — обратная совместимость.
- **Блокирующая модель**: правка → новая `pending`-версия; `description` не
  меняется до апрува противоположной стороны.
- **Одна pending за раз**: частичный unique-индекс
  `CREATE UNIQUE INDEX ... ON task_tz_versions (task_id) WHERE status = 'pending'`.
- Создание задачи с непустым `description` = версия 1, сразу `approved`
  (`proposed_by = resolved_by = author`).
- **Backfill существующих задач (6.4a):** миграция создаёт для каждой уже
  существующей задачи с непустым `description` версию №1 `approved`
  (`proposed_by = resolved_by = author_id`, `created_at = task.created_at`).
  Иначе у старых задач не будет «действующей» версии и история начнётся пусто.
- `version_no` — `max(version_no)+1` в рамках задачи (вычисляется в RPC).
- Индексы: `task_id`, `(task_id, status)`.

### Таблица `task_comments` (обсуждение — гибрид лента/вопросы)

| Колонка | Тип | Назначение |
|---|---|---|
| `id` | uuid PK default gen_random_uuid() | |
| `task_id` | uuid NOT NULL → project_tasks(id) ON DELETE CASCADE | |
| `author_id` | uuid NOT NULL → profiles(id) | |
| `body` | text NOT NULL | текст сообщения |
| `is_question` | bool NOT NULL default false | помечено как вопрос |
| `resolved` | bool NOT NULL default false | вопрос закрыт (только при is_question) |
| `resolved_by` | uuid NULL → profiles(id) | |
| `resolved_at` | timestamptz NULL | |
| `created_at` | timestamptz NOT NULL default now() | |

- Счётчик открытых вопросов задачи = `count(*) WHERE is_question AND NOT resolved`.
- `resolved` имеет смысл только при `is_question = true` (обычное сообщение не
  «решается»).
- Индексы: `task_id`, `(task_id, is_question, resolved)`.

## 3. Права (RLS)

Обе таблицы наследуют видимость от родительской задачи. Поскольку RLS на
`project_tasks` уже описывает «доступ от проекта, иначе личная» (см. 6.4a §3),
для дочерних таблиц предикат:

```
EXISTS (SELECT 1 FROM project_tasks t WHERE t.id = task_id AND <видимость задачи>)
```

где `<видимость задачи>` — тот же предикат, что в SELECT-политике `project_tasks`.
Чтобы не дублировать длинный предикат, ввести SQL-функцию-хелпер
`can_access_task(p_task_id uuid) returns boolean` (STABLE, SECURITY DEFINER),
переиспользующую существующие `can_access_project_comments` / `is_admin` и т.п.
Её же использовать в RPC §4.

- **task_tz_versions**
  - SELECT: `can_access_task(task_id)`.
  - INSERT/UPDATE — **только через RPC** (`SECURITY DEFINER`), прямые мутации
    политиками не разрешаем (нужен контроль «одна pending», «противоположная
    сторона», синхронизация `description`). DELETE запрещён.
- **task_comments**
  - SELECT: `can_access_task(task_id)`.
  - INSERT (with_check): `is_approved() AND author_id = auth.uid() AND can_access_task(task_id)`.
  - UPDATE — **запрещён прямыми мутациями** (RLS не ограничивает колонки, иначе
    можно было бы переписать чужой `body`). Закрытие вопроса (`resolved`) — только
    через RPC `resolve_question` (§4). DELETE запрещён в MVP.

## 4. Доступ к данным (RPC + мутации)

### Версии ТЗ — все через RPC (SECURITY DEFINER, дублируют предикат доступа)

- **`get_task_versions(p_task_id uuid)`** — STABLE. Список версий + имена
  `proposed_by`/`resolved_by`. Сортировка по `version_no`.
- **`propose_tz_version(p_task_id uuid, p_content text)`** — VOLATILE.
  Проверки: доступ; вызывающий — автор или исполнитель задачи, либо
  редактор проекта/админ; нет открытой `pending` (иначе ошибка `tz_pending_exists`).
  Приоритет веток (важно — автор/исполнитель ВСЕГДА идут через двусторонний апрув):
  - Если у задачи **есть исполнитель** (`assigned_to IS NOT NULL`) И вызывающий —
    автор ИЛИ исполнитель: создаётся `pending`-версия, `description` не трогаем
    (двусторонний апрув обязателен, даже если вызывающий по совместительству
    редактор проекта/админ).
  - Иначе (нет исполнителя, ИЛИ вызывающий — посторонний редактор проекта/админ,
    не являющийся ни автором, ни исполнителем): версия сразу `approved`,
    `description` обновляется немедленно.
  Возвращает созданную версию.
- **`approve_tz_version(p_version_id uuid)`** — VOLATILE. Проверки: версия
  `pending`; апрувит **противоположная сторона** (не `proposed_by`; и при этом
  одна из сторон задачи). Действие: `status='approved'`, `resolved_by`,
  `resolved_at`, синхронизация `project_tasks.description = content`.
- **`reject_tz_version(p_version_id uuid)`** — VOLATILE. Аналогично, но
  `status='rejected'`, `description` не меняется. Освобождает «одну pending».

### Смена статуса задачи (апрув сдачи)

- Переход в «**Готово**» разрешён **только автору** задачи. Исполнитель доводит
  максимум до «**На проверке**». Реализация: RPC `set_task_status(p_task_id, p_status)`
  (SECURITY DEFINER) с проверкой роли, либо ужесточение существующей UPDATE-политики
  `project_tasks` предикатом на переход в «Готово». **Выбор реализации — на этапе
  плана** (предпочтительно RPC для единообразия с версиями и чистоты проверок).
  Telegram `task_status` уже шлётся в 6.4a — переиспользуем.

### Комментарии

- **`get_task_comments(p_task_id uuid)`** — STABLE, SECURITY DEFINER. Лента +
  имя `author_id` + флаги. Сортировка по `created_at`.
- **Добавление** — прямой `client.from('task_comments').insert(...)` под RLS.
- **`resolve_question(p_comment_id uuid, p_resolved bool)`** — VOLATILE,
  **только SECURITY DEFINER RPC** (прямой UPDATE на `task_comments` запрещён, см. §3).
  Меняет лишь `resolved`/`resolved_by`/`resolved_at`, только для
  `is_question`-сообщений; право: автор задачи, исполнитель или admin.

## 5. Realtime

- `project_tasks` уже в `supabase_realtime` (с 6.4a). Новые таблицы **не**
  добавляем в publication.
- Фронт: `supabase.channel('project_tasks').on('postgres_changes', {event:'*',
  schema:'public', table:'project_tasks'}, ...)` → мёрж INSERT/UPDATE/DELETE в
  состояние списка/доски. Живое движение карточек по колонкам, появление новых
  задач, смена статусов.
- Внутри **открытой** карточки: версии и комментарии грузятся refetch при
  открытии; при Realtime-событии по `id` открытой задачи — повторный refetch
  версий/комментариев (точечно, без подписки на дочерние таблицы).
- Architecture точка из 6.4a (§5 «список в одном состоянии») уже это допускает —
  переделок MVP не требуется.
- **RLS-приватность подписки (проверить!):** `postgres_changes` на self-hosted
  Supabase обязан уважать RLS `project_tasks`, иначе приватные/чужие задачи
  «протекут» подписчику. На этапе плана проверить, что Realtime отдаёт строки с
  учётом RLS (Realtime authorization / RLS на канале); подписку вести под JWT
  пользователя, не под anon/service. Если RLS на потоке не применяется — фильтровать
  на клиенте по доступным задачам (fallback), но это не отменяет проверку.

## 6. Telegram-relay

**Проблема:** Edge Function `telegram-notify` (локальный Supabase на ПК) не может
достучаться до `api.telegram.org` — точечно заблокирован у провайдера ПК
(см. память проекта). VPS в Москве (193.124.130.236) — кандидат на relay.

**Решение:**
1. **Шаг 1 (verify):** проверить с VPS, открыт ли `api.telegram.org`
   (`curl -sS -o /dev/null -w '%{http_code}' https://api.telegram.org`).
   Если **закрыт и там** — relay не спасёт; эскалация владельцу (альтернатива —
   другой релей-хост). Дальнейшие шаги выполняются только при успехе.
2. **Edge Function** `telegram-notify`: вынести базовый URL в env
   `TELEGRAM_API_BASE` (default `https://api.telegram.org`). На ПК выставить
   `https://193-124-130-236.sslip.io/tg`.
3. **Caddy на VPS**: маршрут `handle_path /tg/* { reverse_proxy https://api.telegram.org { header_up Host api.telegram.org } }`
   (точная директива — на этапе плана; суть: прокси на Telegram Bot API с
   корректным Host/SNI). Конфиг — `/etc/caddy/Caddyfile`, перезапуск `systemctl reload caddy`.
   - **Маршрут `/tg/*` объявить РАНЬШE fallback `→ :8080`** (порядок matcher'ов в
     Caddyfile важен, иначе уйдёт в заглушку).
   - **Токен бота попадает в URL-путь** (`/tg/bot<TOKEN>/...`) → отключить
     access-log для `/tg/*` (`log { output discard }` в этом handle), чтобы токен
     не оседал в логах VPS.
4. Токен бота уже лежит в `volumes/functions/telegram-notify/config.json` (не в git).

**Новые типы уведомлений** (резолв адресатов внутри Edge Function под service_role,
фильтр `notif_task` + наличие `telegram_chat_id`):
- `task_tz_proposed` → противоположной стороне (кому ждать апрува).
- `task_tz_approved` / `task_tz_rejected` → тому, кто предложил версию.
- `task_question` → другой стороне задачи (исполнитель спросил → автору, и наоборот).
- (`task_status` для сдачи — уже есть в 6.4a.)

Вызовы с фронта — best-effort (как в 6.4a): после `propose_tz_version` →
`task_tz_proposed`; после approve/reject → соответствующий тип; после добавления
комментария с `is_question` → `task_question`.

## 7. UI (src/App.jsx, TaskModal)

Все правки — в существующих компонентах слоя задач 6.4a, без новых страниц.

- **Поле ТЗ**: текущий `description` (действующая версия) + кнопка «Изменить ТЗ» →
  textarea → «Предложить изменение» (вызов `propose_tz_version`).
- **Баннер pending**: если есть `pending`-версия — плашка «Предложены изменения ТЗ»
  с **построчным diff** (действующее `approved` vs `pending.content`) и кнопками
  «Принять» / «Отклонить» — **видны и активны только противоположной стороне**;
  предложивший видит «Ожидает подтверждения».
- **Секция «История ТЗ»**: список версий (`get_task_versions`) — номер, автор,
  статус, дата; клик по версии → diff **против предыдущей `approved`-версии**
  (`rejected`-версии видны в списке как отклонённые, но в базе диффа не участвуют —
  сравниваем только то, что было действующим).
- **Секция «Обсуждение»**: лента `get_task_comments` (аватар-инициал, имя, время,
  текст); поле ввода + чекбокс «Это вопрос»; бейдж «N открытых вопросов» в шапке
  секции; на сообщениях-вопросах кнопка «Пометить решённым» (`resolve_question`).
- **diff построчно**: чистая JS-функция (LCS по строкам) в утилитах, без новых
  npm-пакетов. Удалённые строки — красным, добавленные — зелёным.
- Адаптеры snake↔camel для новых сущностей по образцу `taskDbToJs`.

## 9. Критерии приёмки 6.4b

1. Правка ТЗ создаёт `pending`-версию; действующий `description` не меняется до
   апрува; одна pending за раз (вторая правка отклоняется с понятной ошибкой).
2. Апрув противоположной стороной переводит версию в `approved` и обновляет
   `description`; reject — оставляет `description`, освобождает pending.
3. Без исполнителя / привилегированный редактор — правки применяются сразу
   (`approved`), без блокировки.
4. Построчный diff корректно показывает добавленные/удалённые строки (действующая
   `approved` vs `pending`; в истории — версия против предыдущей `approved`).
5. Переход в «Готово» доступен только автору; исполнитель — максимум «На проверке».
6. Обсуждение: сообщения добавляются и видны всем с доступом к задаче; флаг
   «вопрос» увеличивает счётчик открытых; «решено» — уменьшает.
7. Видимость версий и комментариев соответствует доступу к задаче (проверка под
   двумя пользователями: посторонний не видит).
8. Realtime: при изменении задачи у второго пользователя карточка/статус на
   доске обновляются без перезагрузки. **При этом подписка не отдаёт задачи вне
   доступа пользователя** (приватные/чужие не «протекают» через Realtime —
   проверить под двумя пользователями).
9. Telegram: при наличии доступа к `api.telegram.org` с VPS — приходят
   `task_tz_proposed` / `task_tz_approved` / `task_tz_rejected` / `task_question`
   адресатам с `notif_task=true` и привязанным чатом; инициатор себе не дублирует.
   (Если TG недоступен и с VPS — пункт помечается заблокированным, эскалация.)
10. Сборка фронта проходит; деплой через `deploy/nextcloud/deploy-web.sh`.
