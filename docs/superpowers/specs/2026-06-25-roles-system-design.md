# Система ролей и доступа — дизайн (2026-06-25)

Дата: 2026-06-25. Стадия: дизайн (brainstorm с владельцем завершён, одобрен «да ок»).
Метод: brainstorming (4 уточняющих вопроса → решения → дизайн → approval).
Поглощает и расширяет `client-role-design.md` (роль заказчика — частный случай этой системы).

## 1. Контекст
Из эксплуатации: нужно управлять, кто есть кто на сайте, с самостоятельной регистрацией.
Текущее состояние (сверено с живым кодом):
- `profiles.role` CHECK ('admin','user'), `is_admin()` — ~24 места в коде.
- Регистрация: autoconfirm + админ-`approved`; вход по email (Supabase auth).
- Роль заказчика ФАЗА 1 готова: `clients.user_id` привязка + `get_my_client_projects` + раздел «Мои заказы».
- ⚠ `client-role-design §2.3` СОЗНАТЕЛЬНО не вводил роли — этот спек ОТМЕНЯЕТ то решение (вводим явные роли).

## 2. Решения владельца (brainstorm 2026-06-25)
1. Роли: **пользователь(employee) / заказчик(client) / посетитель(visitor)**; мультироль (несколько сразу).
2. Назначение ролей — в UI админом («кто есть кто»).
3. Регистрация: юзер сам выбирает роль; **вход логин+пароль, без почты**.
4. Одобрение: админ назначает роль; **посетитель заходит сам** (без одобрения).
5. Посетитель = **демо-обзор** системы (витрина).
6. Сброс пароля — **только через админа**.
7. Мультироль — **переключатель вида** в шапке.
8. Самрегистрация — **открытая, потом одобрение** (visitor сразу).

## 3. Модель данных
- `public.user_roles(user_id uuid FK auth.users ON DELETE CASCADE, role text CHECK in ('employee','client','visitor'), PRIMARY KEY(user_id, role))`. Мультироль = несколько строк на user_id.
- `admin` НЕ мигрируем в user_roles — остаётся `profiles.role='admin'`/`is_admin()` (не ломаем ~24 места).
- `profiles.username text UNIQUE` — логин без почты.
- Хелпер `has_role(p_uid uuid, p_role text) RETURNS boolean` (STABLE SECURITY DEFINER, `set search_path=public,pg_temp`).
- Миграция данных: approved `user` → `employee`; привязанные заказчики (`clients.user_id IS NOT NULL`) → дополнительно `client`.

## 4. Вход без почты — выбран ВАРИАНТ А (мягкий, без миграции существующих)
- Регистрация/вход по `username`; под капотом синтетический `username@klimat.local` для Supabase-auth (он требует email — честный обход).
- **Существующие email-аккаунты сохраняют email-вход** (вариант А); новые — username. Email-поле скрыто из новых форм.
- Сброс пароля — только админ (service_role `updateUserById`; механизм admin-reset уже есть).

## 5. Доступ по ролям
- `employee` — как сейчас (проекты/задачи/etc), гейт `approved`.
- `client` — портал «Мои заказы» по привязке `clients.user_id` (фаза 1) + фазы 2-3 (переписка/файлы/уведомления — отдельный трек, см. IDEAS).
- `visitor` — **демо-витрина: статичный мок во фронте**, реальные RPC не вызываются (ноль утечки реальных данных). `approved` сразу.
- `admin` — суперправа, как сейчас.

## 6. Мультироль — переключатель вида
- Ролей ≥2 (напр. `employee`+`client`) → тумблер в шапке: «Кабинет сотрудника ↔ Портал заказчика». Текущий вид в localStorage.
- Чистый `visitor` — отдельный демо-вид.

## 7. Регистрация + одобрение
- Форма: `username`, пароль, выбор роли (employee/client/visitor).
- `visitor` → `approved=true` сразу (видит демо). `employee`/`client` → `approved=false`, ждут.
- Админка «Пользователи»: одобрение + чекбоксы ролей (мультивыбор) = «назначить кто есть кто».

## 8. Фазы реализации
- **Ф1:** `user_roles` + миграция + `has_role` + админка-назначение ролей + переключатель вида.
- **Ф2:** вход по `username` без почты (формы входа/регистрации, синтетический email).
- **Ф3:** посетитель + демо-режим (мок-витрина).
- (роль заказчика фазы 2-3 — отдельный трек из IDEAS.)

## 9. Верификация (на живой БД, BEGIN…ROLLBACK, эмуляция через `request.jwt.claims`)
- Мультироль: юзер с `employee`+`client` видит и кабинет, и портал (переключатель).
- `visitor`: реальные RPC не отдают данных; видит только демо-мок; `approved` сразу.
- Регрессия: `is_admin()` и ~24 места целы; существующий email-вход работает.
- Самрегистрация `employee`/`client` → `approved=false` до одобрения админом.
- `set`-роли под не-админом → отказ.

## 10. Вне scope (YAGNI)
- Самовосстановление пароля (только админ). Биллинг. Соцвход. Email-уведомления (почты нет).

## 11. Риски и ловушки
- ~24 места `role==='admin'` — admin вне `user_roles`, проверить регрессию после миграции.
- Синтетический email — уникальность `username`, коллизии, валидация (латиница/цифры).
- Демо-мок посетителя — поддерживать соответствие реальному UI при изменениях.
- Конфликт с `client-role §2.3` — этот спек главнее (вводим роли осознанно).
