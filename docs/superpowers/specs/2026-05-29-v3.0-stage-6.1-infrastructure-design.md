# daniil-dashboard v3.0 — Этап 6.1: Инфраструктура (дизайн)

Дата: 2026-05-29
Статус: одобрен владельцем, в реализации
Источник требований: ТЗ v3.0, разделы 3 и 6.1

## Контекст и принятые решения

| Вопрос | Решение | Обоснование |
|---|---|---|
| Где сервер | **Текущая машина** (Windows 11 Pro, 64 ГБ RAM, 12 ядер) | подтверждено владельцем; ресурсов с запасом |
| Тип IP | **CGNAT (серый)** | трассировка: `192.168.1.1 → 10.228.8.1 → 172.31.253.5 → 10.2.1.16 → публичный`. Доступа к роутеру нет → проброс портов невозможен |
| Внешний доступ | **Cloudflare named tunnel** (исходящий, бесплатный) | CGNAT + нет роутера → нужен исходящий туннель; старт бесплатно, миграция на российский VPS при фильтрации CF |
| Постоянный URL | **домен `.ru` + Cloudflare** (~199 ₽/год) | named tunnel требует домен в DNS Cloudflare; quick tunnel даёт нестабильный URL |
| Размещение бэкенда | весь бэкенд **дома**, туннель наружу (вариант A ТЗ) | по выбору владельца |

## Целевая схема

```
[Браузер / PWA телефона]
        │ HTTPS (валидный сертификат от Cloudflare)
        ▼
   [Cloudflare edge]        ← TLS терминируется здесь
        │ исходящий туннель (ПК инициирует соединение ИЗНУТРИ)
        ▼
 [cloudflared (служба Windows)]
        │ ingress по路ям
        ├──→ domain.ru      → localhost:8080 → [nginx: собранный React]
        └──→ api.domain.ru  → localhost:8000 → [Kong: API-шлюз Supabase]
                                                      │
                              Docker compose (self-hosted Supabase):
                              postgres · gotrue · postgrest · realtime ·
                              storage · kong · studio · meta
```

Наружу не открыт ни один порт. CGNAT и отсутствие доступа к роутеру нерелевантны —
`cloudflared` держит исходящее TLS-соединение к Cloudflare edge.

## Компоненты

| Компонент | Роль | Примечание |
|---|---|---|
| Docker Desktop (WSL2) | движок контейнеров | WSL2 на машине уже есть |
| self-hosted Supabase (`supabase/docker`) | БД, авторизация, REST, Realtime, Storage | замена облака (цель №1) |
| nginx-контейнер | отдаёт собранный React | TLS не нужен (даёт Cloudflare) |
| cloudflared | туннель наружу | служба Windows, автозапуск |

Не нужны (упрощение относительно ТЗ из-за выбора Cloudflare): Caddy/nginx для TLS,
Let's Encrypt, DDNS — Cloudflare берёт TLS и DNS на себя.

## Безопасность

- Через туннель публикуются **только** фронт и Kong (`:8000`).
- **Supabase Studio и Postgres — localhost-only**, в туннель не выводятся.
- Секреты (JWT-секрет, пароль Postgres, anon/service ключи) — в `.env`,
  **не коммитятся** (в `.gitignore`).

## Автозапуск

- Docker Desktop — автостарт при входе; контейнеры `restart: unless-stopped`.
- cloudflared — служба Windows (`cloudflared service install`), поднимается до входа.

## Порядок реализации

1. Установить Docker Desktop (WSL2 backend).
2. Развернуть self-hosted Supabase (`docker compose`, сгенерировать свои секреты/JWT).
3. Локальная проверка: Studio на `localhost`, API на `:8000` отвечает.
4. nginx-контейнер со страницей-заглушкой (health).
5. Купить домен `.ru`, делегировать NS на Cloudflare. *(действие владельца)*
6. Установить cloudflared, создать named tunnel, ingress-правила.
7. Настроить автозапуск.
8. Проверка критерия готовности.

## Критерий готовности (из ТЗ 6.1)

- Локальный Supabase отвечает (Studio + API).
- Сайт-заглушка открывается по домену **снаружи, без VPN** (проверка с телефона по мобильному интернету).
- Всё поднимается само после перезагрузки ПК.

## Ревизия 2026-05-29: Cloudflare Tunnel → российский VPS + FRP

При проверке quick-tunnel Cloudflare в сети владельца **не держится**:
- QUIC (UDP 7844) полностью блокируется провайдером;
- http2-туннель регистрируется, но соединение рвётся через секунды
  (`530 origin unregistered`, 5/5 проверок снаружи провалены).

Причина: исходящие соединения к зарубежной инфраструктуре (Cloudflare,
несмотря на московский edge) у владельца нестабильны без VPN — та же
проблема, ради которой затевается v3.0. Бесплатный иностранный туннель
её не обходит. Решение принято владельцем: **дешёвый российский VPS**
как точка входа.

### Новая схема точки входа

```
[Браузер/PWA] --HTTPS--> [Российский VPS, белый IP]
                          Caddy :443 (TLS, Let's Encrypt по <ip>.sslip.io)
                                 │
                          frps (FRP-сервер)
                                 ▲ исходящий туннель (ПК инициирует, внутри РФ — стабильно)
                                 │
                          frpc (FRP-клиент) на ПК (WSL)
                                 ├─→ 127.0.0.1:8080 (nginx web)
                                 └─→ 127.0.0.1:8000 (Kong API)
```

- Домен не нужен: TLS через `<vps-ip>.sslip.io` + Let's Encrypt.
- VPS — тонкий форвардер (~170 ₽/мес), данные и нагрузка остаются на ПК.
- cloudflared удаляется из схемы; Docker+Supabase+nginx без изменений.

### Транспортный урок

В сети владельца UDP/QUIC к зарубежным сервисам режется; длинные TLS к
зарубежным edge нестабильны. К российскому VPS (FRP по TCP/TLS) — стабильно.
